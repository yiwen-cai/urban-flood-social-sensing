# 协作规范

本项目由三名成员并行完成。GitHub 仓库是代码、接口和正式交付状态的唯一事实源；聊天或云文档中的决定必须回写仓库才能生效。

## 分支与合并

- `main` 始终保持可展示、可回归，不直接在 `main` 上开发；
- 成员 A 使用 `lab1/data-*`，成员 B 使用 `lab2/analysis-*`，成员 C 使用 `lab3/dashboard-*`；
- 一项交付对应一个 Pull Request，标题注明 D01–D12 编号；
- PR 必须由实施方案指定的下游验收人复核后才能合并；
- 接口修改必须在 PR 中说明字段变化、影响模块、迁移方式和回滚方式。

## 交接标准

交接遵循：生产者自检、接收人复验、下游成功读取。仅完成代码但下游无法消费产物，不算完成。

每个 PR 至少说明：

- 对应交付物编号；
- 正式产物路径；
- Schema 和流水线版本；
- 使用的验证命令及结果；
- 已知问题与回滚方式；
- 是否涉及数据、隐私或标签语义变化。

## 数据与安全

- 禁止提交 `data/raw/`、未脱敏文本、真实联系方式或精确住宅地址；
- 禁止提交 `.env`、API Key、Cookie、Token 或个人配置；
- `tests/fixtures/` 只能包含人工构造或完成审计的脱敏样例；
- 公开展示前必须完成来源、许可、隐私和事实可追溯性检查；
- 大型数据文件和演示视频使用受控云盘或 GitHub Release，不进入 Git 历史。

## 每日节奏

每天结束前合并已验收的交付，并由三人共同运行一次离线 fixture 回归。Day 3 冻结功能，Day 4 只修复阻塞性问题、完善汇报和演练。
